上周，部门领导做了一次 AI 编程分享，这次确实让我有点震动。

让我印象最深的，并不是用 AI 写了个多炫的 demo，而是他在分享里展示的，是在用 AI 真正做产品。而且这个产品并不是什么简单的小工具，而是一个能够提高效率的调试软件，有着相当的复杂度。

更让我意外的是，他用的工具，正是最近很火的 OpenClaw。说实话，我前段时间也折腾过 OpenClaw，但更多停留在挺有意思的层面，从来没有想过，它还能被这样用，放进一套工程化流程里去用。

通过工程化的思路，结合 OpenClaw 本身的上下文和记忆能力，把整个开发过程拆成结构化的阶段，有分工、有约束，让它逐步参与到设计、拆解、实现的各个环节里。最终呈现出来的效果，已经很接近 Claude Code 这种级别的开发体验。

这件事给我的一个直接感受是：

> 工具本身重要，但远没有怎么用好重要。

如果只是把 AI 当成补全代码的工具，那它的上限也就在那里，但如果有一套清晰的软件工程思维，能把工具的能力嵌入到流程里，它能发挥出来的效果会完全不一样。

也正因为这个契机，我决定重新认真用一用 OpenCode。这一次，不再只是浅尝辄止，而是尝试把它融入到开发流程，真正提高工作中的开发效率。

过去我对 AI 编程的评价大概是三点：

* 写函数很强
* 写 demo 很快
* 一旦项目变大，就开始失控

你让它写一个文件，它很强，但你让它维护一个项目，它就开始遗忘、冲突、乱改文件。这也是我一直将 AI 当做编程助手，而没有应用到项目中的原因。

## 二、OpenCode

简单来说，OpenCode 是一个开源的 AI 编程代理（coding agent），可以在终端或 IDE 中参与开发工作。它的核心形态是一个 CLI 工具，但通过插件或扩展，也可以比较自然地集成到 VS Code 等开发环境中使用。

在 AI 编程领域，如果提到当前的标杆产品，很多人首先想到的还是 Claude Code。一方面，Claude 系列模型在代码理解和生成方面表现一直非常突出；另一方面，Claude Code 在工程化能力上的探索也走在前面，比如：

* skills（能力模块化）
* agent（任务执行主体）
* 多阶段工作流

这些思路并非全部起源于 Claude，但确实是在 Claude Code 这类产品中被系统化、产品化，并开始被更多人熟悉和使用。

目前业界已经有不少开发者在使用 Claude Code 进行实际开发。

而 OpenCode，可以理解为在这一方向上的一个开源实践。虽然开源替代方案在整体体验上，往往还难以完全追平成熟的商业产品，但 OpenCode 也有它非常明确的优势：

1. 内置了一些可免费使用的模型，比如 MiniMax M2.5、Big Pickle、MiMo V2 Pro 等。同时，OpenCode 本身并不绑定任何特定模型，而是支持接入多家模型提供商，也允许用户自定义模型配置。这种模型解耦的设计，带来的好处很直接，可以根据不同任务灵活切换模型，日常开发优先使用免费模型，控制成本，不会有 tokens 焦虑。
2. 兼容 Claude Code 的生态，例如可以自动读取和复用 Claude Code 中的 CLAUDE.md、SKILLS、COMMANDS 等配置。这意味着你之前在 Claude Code 中已经打磨好的能力，可以在 OpenCode 中直接沿用，而不需要从头再来。
3. OpenCode 是开源的，这一点其实非常关键。相比之下，一些商业化工具，比如 Claude 经常会封中国开发者的账号，带来不可控因素。

> **OpenCode 是一个“可以被编排”的 AI 开发执行层。**

---

## 1. 它和传统 AI 编程工具的差别

很多人接触 AI 编程，最早用的是：

* Copilot
* ChatGPT
* 各种代码助手

这些工具有一个共同特点：

> **以“补全”和“对话”为核心**

而 OpenCode 的思路不太一样，它更接近：

> **让 AI 变成一个可以执行任务的 Agent**

比如它可以：

* 在项目中读取和理解代码结构
* 执行命令（构建、测试等）
* 按步骤完成任务，而不是一次性生成

这也是为什么它更容易和“工程流程”结合。

---

## 2. 最大的优势：模型解耦

OpenCode 一个非常现实、但又非常关键的设计是：

> **它不绑定任何具体模型**

你可以接入：

* Claude
* GPT
* Gemini
* 甚至本地模型

官方也提到，它支持几十种不同的模型提供商。 ([OpenCode][1])

这带来的好处非常直接：

* 可以根据任务切换模型
* 可以优先使用免费模型
* 不会被某一家厂商锁死

对于日常开发来说，这一点甚至比“模型强不强”更重要。

---

## 3. 多形态：不仅仅是 CLI

OpenCode 并不是一个单一形态的工具，它同时支持：

* 终端（CLI / TUI）
* IDE 插件
* 桌面应用

也就是说，你可以：

* 在终端里像用 git 一样用它
* 或者嵌入现有开发环境

这一点让它的使用门槛比很多 Agent 工具要低。 ([OpenCode][1])

---

## 4. LSP + 多会话：更接近“开发环境”

OpenCode 还有两个比较偏工程向的能力：

### （1）LSP 支持

它可以自动加载语言服务器（LSP），从而获得：

* 语法信息
* 错误诊断
* 类型提示

这意味着：

> AI 不只是“猜代码”，而是“基于真实语义分析在写代码”

---

### （2）多会话（Multi-Agent）

可以在同一个项目里启动多个会话：

* 一个做实现
* 一个做调试
* 一个做重构

这其实已经接近“多人协作”的模型了。

---

## 5. Plan / Build 模式：开始走向工程化

OpenCode 社区里一个很重要的实践是：

> **Plan → Build 的两阶段开发模式**

也就是：

1. 先分析需求，生成修改计划
2. 再按计划执行代码修改

这点和后面提到的 Superpowers，其实是可以很好结合的。

👉 本质上是在解决一个老问题：

> 不让 AI“想到哪写到哪”

---

## 6. 开源 + 隐私：它能进企业环境

还有一个经常被忽略的点：

> **OpenCode 是开源的，而且默认不存储你的代码和上下文数据** ([OpenCode][1])

这意味着：

* 可以自部署
* 可以接入内网模型
* 更容易进入企业环境

相比一些完全云端的工具，这一点在实际项目里非常重要。

---

## 7. 一个更准确的定位

如果一定要给 OpenCode 一个定位，我会这么说：

> 它不是“更强的 Copilot”，
> 而是一个“可以承载开发流程的 AI 执行层”。

也正因为如此，它本身并不会直接带来“工程化”。

真正让它发挥价值的，是：

* 上层的 workflow（比如 Superpowers）
* 以及你自己的工程方法

---

这一段和你后面的 Superpowers 是可以自然衔接的：

👉 OpenCode = 执行层
👉 Superpowers = 方法论 / 流程层

这样整篇文章的结构会非常清晰，也更有说服力。

[1]: https://opencode.ai/zh?utm_source=chatgpt.com "OpenCode | 开源 AI 编程代理"



我试了一圈之后，最后留下的是 OpenCode。

原因很现实：它解决的是“可控性”，而不是“能力上限”。

### 1. 模型不是重点，调度才是

OpenCode 最大的价值，不是它绑定了某个模型，而是：

> **它把模型当成“可替换资源”**

比如我现在主要用的是：

* MiniMax M2.5 Free（免费、量大）
* 其他模型作为补充（复杂任务）

对于日常开发来说，这一点非常关键：

* 成本可控
* 不会因为 token 焦虑影响使用
* 可以按任务切模型

---

### 2. 兼容 Claude Code 生态（这点很关键）

OpenCode 支持类似 Claude Code 的：

* skills
* 插件（plugin）

这意味着什么？

👉 AI 不再只是“聊天”，而是开始具备**工具能力**

比如：

* 读取项目结构
* 执行命令
* 分阶段生成文档（plan / design / tasks）

这一步，其实已经从“Copilot”走向“Agent”。

---

## 三、Superpowers 插件：真正补上工程化的一块

如果说 OpenCode 提供的是“底座”，
那 Superpowers 插件做的事情是：

> **给 AI 加了一套“做事的方法论”。**

我用下来，它最有价值的不是功能，而是这几件事：

---

### 1. 强制分阶段，而不是一把梭

传统 AI 编程是这样的：

> “帮我写一个 XXX”

然后一坨代码出来。

Superpowers 的思路是：

1. `/plan` —— 先做规划
2. `/design` —— 再做设计
3. `/tasks` —— 拆任务
4. `/implement` —— 执行

这看起来很“重”，但实际效果是：

👉 **上下文稳定了，输出质量反而更高**

---

### 2. 文档驱动，而不是对话驱动

生成的内容会落到文件里，比如：

* `architecture.md`
* `design.md`
* `tasks.md`

这件事的意义在于：

> AI 的“记忆”从上下文，变成了“项目资产”。

这其实非常接近真实开发流程。

---

### 3. 可重复，而不是一次性

很多 AI 生成的东西是“一次性的”。

但有了这些文档之后：

* 可以反复修改
* 可以让 AI 基于已有设计继续做
* 可以团队共享

👉 这一步，才真正开始像“工程”。

---

## 四、实战：用它做一个 OpenClaw Desktop App

光说不练没意义，我用一个实际项目跑了一遍流程：
👉 做一个 OpenClaw Desktop App（桌面客户端）

### 1. 第一步：规划（/plan）

我只做了一件事：

> 描述我要做一个桌面版 OpenClaw 客户端，支持基础对话、插件扩展

AI 输出了：

* 模块划分
* 技术选型（Electron / Qt / WebView 等）
* 基本架构

这一步最重要的不是“对不对”，而是：

👉 **把问题空间收敛了**

---

### 2. 第二步：设计（/design）

在 plan 基础上继续细化：

* UI 结构
* 插件系统设计
* 与模型交互方式

这一步我会人工介入，做一些取舍。

经验是：

> **不要完全相信 AI，但要利用它的“穷举能力”**

---

### 3. 第三步：任务拆解（/tasks）

这里开始变得很实用了：

* 每个模块变成具体任务
* 有执行顺序
* 有依赖关系

基本上已经接近一个简单的项目管理工具了。

---

### 4. 第四步：实现（/implement）

这一步才是传统意义上的“写代码”。

但有了前面的结构之后：

* 上下文更清晰
* 代码风格更统一
* 返工明显减少

---

## 五、一些真实感受（不是宣传）

最后说点更实际的。

### 1. 它还远没到“自动写完项目”的程度

如果你期待的是：

> 输入一句话 → 出一个完整产品

那现在还是不现实。

---

### 2. 但已经可以“放大个人能力”

更准确的定位是：

> **从“写代码工具”，变成“开发放大器”**

适合的场景：

* 新项目起步
* 不熟悉的技术栈
* 做 PoC / 工具类项目

---

### 3. 工程化思维，依然是核心

一个很有意思的结论是：

> 用 AI 写工程，反而更考验工程能力。

因为你需要：

* 判断结构是否合理
* 控制复杂度
* 做技术选型

AI 只是执行者。

---

## 六、小结

这一轮折腾下来，我最大的感受是：

> AI 编程真正的分水岭，不在模型，而在“有没有工程体系”。

OpenCode + Superpowers 的组合，至少让我看到了一个方向：

* 模型负责能力
* 插件负责工具
* 流程负责约束

三者结合，才开始接近“能用”的状态。

---

如果你之前对 AI 编程也有类似的怀疑，不妨换个思路：

不要问“它能不能替你写代码”，
而是问：

> **它能不能成为你工程体系的一部分。**

这个问题的答案，已经开始变得有意思了。
